# Forge — 三日健身 App · Design Spec

> Push / Pull / Legs 課表 app，依 %1RM 自動生成重量、逐組追蹤、PR 偵測、身體組成紀錄。
>
> 本 spec 是 `design_handoff_forge/README.md` 的 production 實作定義，含經過網路研究微調的菜單生成公式。

## 4. Data Model

完全依 README §Data Model，但新增一張表支援 Option B 公式（RIR autoregulation）：

### 新增：`exercise_feedback`（每次 workout，每個 trackable 動作最多一列）
```sql
CREATE TABLE exercise_feedback (
  workout_id  bigint REFERENCES workouts(id) ON DELETE CASCADE,
  exercise_id text NOT NULL,
  rir         smallint CHECK (rir BETWEEN 0 AND 5),  -- nullable if skipped
  PRIMARY KEY (workout_id, exercise_id)
);
```

粒度說明：RIR 在 Finish screen 以「每個 trackable 動作填一次」的方式捕捉（代表該動作最重工作組的主觀感受），不是 per-set。

### 新增：`user_progression_state`（記錄 double progression 狀態）
```sql
CREATE TABLE user_progression_state (
  user_id     uuid REFERENCES users(id) ON DELETE CASCADE,
  exercise_id text NOT NULL,
  scheme      text NOT NULL,              -- strength / power / hypertrophy / pump
  current_pct numeric(4,3) NOT NULL,      -- 目前在該 scheme 的 pct band 位置（如 0.800）
  streak      int NOT NULL DEFAULT 0,     -- 連續達 rep range 頂端次數
  updated_at  timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, exercise_id, scheme)
);
```

其餘（`users`, `profiles`, `exercise_maxes`, `last_weights`, `body_logs`, `workouts`, `workout_sets`）完全照 README。

## 5. Workout Generation — Option B（強化公式）

### 5.1 核心原則
1. **Percentage band** 取代固定 pct（用 `current_pct` 追蹤在 band 中位置）
2. **Hybrid 1RM 估算**：reps ≤ 5 用 Epley；6-12 用 Brzycki；> 12 不估
3. **Double progression**：達 rep range 上緣 → 自動建議 band 往上移或 +2.5kg
4. **RIR autoregulation**：Finish screen 可選填 RIR，feed back 到下次建議

### 5.2 Scheme 定義（更新）

| Scheme | Rep range | %1RM band | RIR 目標 | Rest |
|---|---|---|---|---|
| `strength` | 3-5 | 0.80 – 0.85 | 1-2 | 180s |
| `power` | 5-7 | 0.72 – 0.78 | 1-2 | 150s |
| `hypertrophy` | 8-10 | 0.65 – 0.75 | 2-3 | 90s |
| `pump` | 12-15 | 0.50 – 0.60 | 0-2 | 60s |

### 5.3 建議重量演算

```ts
function suggestWeight(userId, exerciseId, scheme): number {
  const ex = EXERCISES[exerciseId];
  const s = SCHEMES[scheme];

  if (ex.trackable) {
    const oneRM = getMax(userId, exerciseId);        // from exercise_maxes
    if (!oneRM) return null;
    const state = getProgressionState(userId, exerciseId, scheme);
    const pct = state?.current_pct ?? s.pctLow;       // 初次從 band 下緣開始
    return round25(oneRM * pct);
  }

  // 非主項：記憶上次
  const last = getLastWeight(userId, exerciseId);
  if (last) return last;
  // fallback：依器材類型推估（與 README / 原 data.js 一致）
  if (ex.equip === 'dumbbell') return Math.max(8, round2(userWeight * 0.15));
  if (ex.equip === 'bodyweight') return null;
  return null;
}
```

### 5.4 1RM Hybrid 估算

```ts
function estimate1RM(weight: number, reps: number): number | null {
  if (reps < 1) return null;
  if (reps === 1) return weight;
  if (reps <= 5)  return weight * (1 + reps / 30);         // Epley
  if (reps <= 12) return weight / (1.0278 - 0.0278 * reps); // Brzycki
  return null;                                              // > 12 reps: 不估
}
```

應用時機（Finish screen）：對每個 trackable 做過的動作，找「最佳工作組」`max(estimate1RM)`，> 舊 max 則建議更新。

### 5.5 Double Progression + RIR 更新邏輯

Finish 時，對每個 (user, exercise, scheme) 呼叫：

```ts
function updateProgression(
  sets: Set[],               // 該 (exercise, scheme) 的所有 sets
  rir: number | null,        // 該動作的 RIR（per-exercise，可能 null）
  scheme: Scheme,
  state: ProgressionState,
): ProgressionState {
  const done = sets.filter(s => s.done && s.reps > 0);
  if (done.length === 0) return state;

  const allHitTop = done.every(s => s.reps >= scheme.repHigh);

  // RIR 規則（有填才生效）— 優先於 streak 判斷
  if (rir !== null) {
    if (rir >= 3 && allHitTop) {
      return { current_pct: min(scheme.pctHigh, state.current_pct + 0.025), streak: 0 };
    }
    if (rir <= 0 && !allHitTop) {
      return { current_pct: max(scheme.pctLow, state.current_pct - 0.025), streak: 0 };
    }
  }

  // Double progression（沒 RIR 也能跑）
  if (allHitTop) {
    const newStreak = state.streak + 1;
    if (newStreak >= 2) {
      return { current_pct: min(scheme.pctHigh, state.current_pct + 0.025), streak: 0 };
    }
    return { ...state, streak: newStreak };
  }

  return { ...state, streak: 0 };
}
```

重點：**pct band 爬到頂 AND 1RM 被 Finish screen 更新** → 下次自然走更高絕對重量；pct 邏輯只在 band 內游走。

### 5.6 Day Suggestion（照 README）
`['push','pull','legs']` 中 `finished_at` 最舊 / 缺席的那天 → 標 `SUGGESTED`。

### 5.7 Day Templates（沿用 data.js，不動）
Push / Pull / Legs 每日 5 slots + 2 core，每 slot 一個 scheme。寫死在 `lib/workout-data.ts`。

## 6. Auth Flow

- 單一 `/auth` 頁（segmented control 切 login/register）
- API: `POST /api/auth/register` / `POST /api/auth/login` / `POST /api/auth/logout`
- 密碼：bcrypt cost 12；client 端也擋明顯錯誤（username regex、長度）
- Session：`iron-session` 存 `{ userId: uuid }` 到 HTTP-only cookie
- Rate limit：**本 MVP 不實作**（VM 外圍 ngrok 不用擔心公開；TODO 標註未來加）
- Register 成功 → 自動 login → redirect `/onboarding`（因為還沒有 `profiles` row）
- Login 成功 → redirect `/today`（若無 profile 則 `/onboarding`）
- Middleware（`src/middleware.ts`）保護 `(app)/*`，未登入 redirect 到 `/auth`

## 7. Screens

完全依 README §Screens（1-8）。實作注意事項：

### 整體 shell
- `(app)/layout.tsx` 提供 440px 最大寬、dark bg、Inter + JetBrains Mono（`next/font/google`）
- 底部 tab bar 只在 `/today`（day selector）與 `/progress` 顯示；workout/finish/auth/onboarding 隱藏

### Workout screen（state 最重的一頁）
- Client component，初始 state 由 server component 預載（pre-filled 建議重量 / reps）
- 本地 state：`sets[]`，每 set `{ weight, reps, done, rir? }`
- 每 N 秒寫一份 draft 到 `localStorage` 做 crash recovery
- 送 finish 時 POST `/api/workouts`（傳完整 sets），server 做：
  - 寫 `workouts` + `workout_sets`
  - 每個 trackable 跑 hybrid 1RM，回傳 `{ exerciseId, oldMax, newEstimate }[]`
  - 寫 `exercise_feedback` for trackable exercises that provided RIR
  - 更新 `last_weights`
- Finish screen 讀 server 回傳的提案，user 可調整 → 另一支 PATCH `/api/maxes`

### RIR 輸入介面
- Finish screen：每個 trackable 動作 block 下方一組 0/1/2/3+ 四顆小 pill（可選、可跳過）
- 記錄到 `exercise_feedback`（per-exercise，非 per-set；代表該動作主觀強度）
